The Database and hence ADB has a feature called AQ (short for advanced queues one presumes) that can be used to provide a pub / sub model for iot events as they arrive.
The IoT documentation details are at https://docs.oracle.com/en-us/iaas/Content/internet-of-things/stream-normalized-queue.htm
Oracle is also developing an AQ subscriber for Node Red, this project is at https://github.com/oracle-samples/node-red-nodes
There is documentation on using python to dequeue the messages here https://python-oracledb.readthedocs.io/en/latest/user_guide/aq.html#
also examples at https://github.com/oracle-samples/oci-iot-samples/tree/main/samples/python/queues

Note that to do this you need to have the rights to do a direct DB connection - see https://github.com/oracle-samples/oci-iot-samples/blob/main/samples/python/query-db/README.md

When the IOT DB Schema is configured as the service is setup the AQ queues for raw (in & out), rejected and normalized data are automatrically setup, all that's needed to handle this data is to have a subscriber identified and then for it to connect

from the python example:

The following queues are available:

| Queue name       | Data type             | Description                      |
| ---------------- | --------------------- | -------------------------------- |
| raw_data_in      | raw_data_in_type      | Incoming raw messages            |
| raw_data_out     | raw_data_out_type     | Outgoing raw messages (commands) |
| normalized_data  | JSON                  | Normalized data                  |
| rejected_data_in | rejected_data_in_type | Rejected incoming messages       |

The `normalized_data` queues is a JSON queue, while the others use an Abstract Data Type
(ADT).
